# Analyze Session

Produce a structured analysis of a Claude Code session from its JSONL log file.

## Inputs

You need exactly one thing: a **path to a JSONL session file**.

### How to get the JSONL path

If you already have the path (e.g., passed by the session-analyzer agent), skip
to the Analysis Workflow.

If you're doing a **self-review** (invoked as a skill mid-conversation):
1. Pick a distinctive phrase from earlier in this conversation
2. Run: `grep -rl "<phrase>" ~/.claude/projects/`
3. Use the matching file

If you were given a **session UUID**:
1. Run: `find ~/.claude/projects -name "<UUID>.jsonl"`

If you were given a **conversation snippet**:
1. Run: `grep -rl "<snippet>" ~/.claude/projects/`

If multiple files match, show the user and ask which one.

## Analysis Workflow

### Step 1: High-level overview with claugs

```bash
claugs show --compact <jsonl_path>
```

Read the output to identify:
- The user's initial request
- Major phase transitions (user messages, long gaps, topic shifts)
- The final outcome

### Step 2: Extract metadata from JSONL

Use Python to extract session metadata:

```python
import json

lines = open('<jsonl_path>').readlines()

# Session ID: from the filename
# Project path: from 'cwd' field on assistant messages
# Branch: from 'gitBranch' field on assistant messages
# Timestamps: first and last 'timestamp' fields

for line in lines:
    obj = json.loads(line)
    if obj.get('type') == 'assistant':
        cwd = obj.get('cwd', '')
        branch = obj.get('gitBranch', '')
        if cwd and branch:
            print(f"Project: {cwd}")
            print(f"Branch: {branch}")
            break

# Git origin URL: run from the project directory
import subprocess
origin = subprocess.run(
    ['git', '-C', cwd, 'remote', 'get-url', 'origin'],
    capture_output=True, text=True
).stdout.strip()
if origin:
    print(f"Origin: {origin}")
```

### Step 3: Extract token usage via claugs

Use the `compute_token_stats` and `compute_context_window_usage` helpers from
the `claude_logs` package (PyPI: `claugs>=0.6.7`). These handle the tricky
parts — deduplication by `message.id` (Claude Code writes multiple JSONL lines
per streaming response, each with the same `message.id`, so naive line-summing
inflates cache totals by up to 70%+) and filtering by line range, UUID range,
or message type for per-phase breakdowns.

```python
from claude_logs import (
    compute_token_stats,
    compute_context_window_usage,
    TokenStatsFilter,
)

path = '<jsonl_path>'

# Whole-session totals
stats = compute_token_stats(path)
total_input = stats.input_tokens
total_output = stats.output_tokens
total_cache_read = stats.cache_read_input_tokens
total_cache_create = stats.cache_creation_input_tokens
api_calls = stats.unique_api_calls

# Per-phase breakdown — use TokenStatsFilter to scope
phase_stats = compute_token_stats(
    path,
    filter=TokenStatsFilter(
        since_uuid="<phase-start-uuid>",   # inclusive
        until_uuid="<phase-end-uuid>",     # inclusive
    ),
)

# Alternative scoping by raw line numbers
phase_stats = compute_token_stats(
    path,
    filter=TokenStatsFilter(line_start=100, line_end=500),
)

# Current context window (most recent assistant turn's input footprint,
# matching what Claude Code's UI displays)
cw = compute_context_window_usage(path)
if cw is not None:
    ctx_total = cw.total  # input + cache_creation + cache_read
    ctx_source_line = cw.source_line
    ctx_source_uuid = cw.source_uuid
```

**What the fields mean:**
- `cache_read_input_tokens` -- bulk of input, served from prompt cache
- `cache_creation_input_tokens` -- new content written to cache for the first time
- `input_tokens` -- tiny uncacheable remainder (typically single-digit per call)
- `output_tokens` -- tokens generated by the model (reported but NOT part of
  the "context window" input footprint)
- **Context window at any call** = `cache_read + cache_create + input_tokens`
  (use `compute_context_window_usage(path).total` — returns the input footprint
  of the most recent non-sidechain assistant turn, excluding sub-agent calls)
- **Peak context** = the largest single-call total across the session. The
  most recent call is usually also the peak because the context grows
  monotonically.
- **Total billed** = cumulative sum across all calls (what `compute_token_stats`
  returns as the four `*_tokens` fields).

**Why claugs instead of hand-rolled extraction**: Claude Code writes a single
Anthropic API response as multiple JSONL lines, one per content block. Each
line has the same `message.id` but a streaming snapshot of `output_tokens`.
Naive line-summing inflates cache totals by 30-70% depending on tool-use
density. `compute_token_stats` groups by `message.id` and takes the field-wise
max, which matches what the Anthropic API actually billed.

**Cost estimation** — the rate table below is manually maintained and can
drift from Anthropic's actual pricing. **Verify before quoting costs in an
analysis**, especially if the session used a non-Opus model.

| Component | Rate (Opus) |
|-----------|-------------|
| Input tokens (uncached) | $15 / MTok |
| Cache read tokens | $1.875 / MTok |
| Cache creation tokens | $18.75 / MTok |
| Output tokens | $75 / MTok |

Rate table last checked: **2026-04-08** (Opus 4.6). Bump this date when you
next verify against <https://www.anthropic.com/pricing>.

### Step 4: Extract turn durations

```python
turns = []
for line in lines:
    obj = json.loads(line)
    if obj.get('type') == 'system' and obj.get('subtype') == 'turn_duration':
        turns.append({
            'ts': obj.get('timestamp', ''),
            'duration_ms': obj.get('durationMs', 0),
            'message_count': obj.get('messageCount', 0),
        })
```

### Step 5: Drill into phases with claugs

Use `--since` and `--until` (without `--compact`) to examine specific time
ranges in detail:

```bash
claugs show --since "YYYY-MM-DDTHH:MM:SSZ" --until "YYYY-MM-DDTHH:MM:SSZ" <jsonl_path>
```

Identify:
- What tools were used and how often
- Where the agent got stuck (repeated attempts at the same thing)
- What breakthroughs occurred and what triggered them
- playlite/ragref usage patterns
- Background task outcomes

### Step 6: Count tool usage

```python
from collections import Counter

tool_calls = Counter()
for line in lines:
    obj = json.loads(line)
    msg = obj.get('message', {})
    if isinstance(msg, dict) and msg.get('role') == 'assistant':
        content = msg.get('content', [])
        if isinstance(content, list):
            for block in content:
                if isinstance(block, dict) and block.get('type') == 'tool_use':
                    tool_calls[block.get('name', 'unknown')] += 1
```

### Step 7: Write the analysis

Write the analysis to `~/.claude-session-analyses/analyses/YYYY-MM-DD-<short-description>.md`

The short description should be derived from the session's primary task (e.g.,
`dwp3-checklist-attributes`, `migrate-wo-completion-test`, `debug-grid-flakiness`).

**Required sections** (in this order):

1. **Overview** -- table with wall clock, active time, tool calls, test runs, deliverables
2. **Timeline & Time Distribution** -- phases with timestamps, durations, activities
3. **What It Accomplished** -- deliverables, files changed, commits
4. **Token Usage Analysis** -- per-phase breakdown table, context window growth, cost estimation
5. **Delegation & Background Work** -- subagents, tasks, background commands, success rates
6. **Friction Points & Resolutions** -- for each: problem, how discovered, what worked, time cost, gap
7. **Tool/System Effectiveness** -- tables: what helped, what caused friction, what was dead weight
8. **Gaps & Recommendations** -- actionable items, each with a pattern ID
9. **Pattern Links** -- list of pattern IDs created or updated

**YAML frontmatter** (must include all these fields):

```yaml
---
session_id: <UUID from filename>
jsonl_path: <absolute path>
project: <cwd from JSONL>
branch: <gitBranch from JSONL>
git_origin: <git remote get-url origin, or omit if not a git repo>
date: <YYYY-MM-DD>
status: raw
api_calls: <count>
output_tokens: <total>
cache_read_tokens: <total>
cache_creation_tokens: <total>
estimated_cost_usd: <float>
active_duration_minutes: <from turn_duration sum>
patterns:
  - <pattern-id-1>
  - <pattern-id-2>
tags:
  - <freeform-tag>
---
```

### Step 8: Create or update pattern files

For each gap/recommendation identified, check if a matching pattern exists in
`~/.claude-session-analyses/patterns/`:

```bash
ls ~/.claude-session-analyses/patterns/
```

- If a pattern file exists with a matching topic: update its `last_seen`, add
  this analysis to its `analyses` list
- If no match: create a new pattern file (see pattern format in the spec at
  `~/.claude-session-analyses/docs/superpowers/specs/2026-04-01-session-analysis-system-design.md`)

Pattern files use this frontmatter:

```yaml
---
id: <kebab-case-id>
title: <Human readable title>
status: observed
first_seen: <YYYY-MM-DD>
last_seen: <YYYY-MM-DD>
resolution_threshold_days: 14
analyses:
  - <analysis-filename.md>
solutions: []
---
```

### Step 9: Report completion

Tell the user:
- Where the analysis was written
- How many patterns were created/updated
- Key findings summary (3-5 bullet points)

## Context

This skill is the core brain of a three-component session analysis system:
1. This skill (analyze-session) -- does the actual JSONL parsing and report generation
2. An agent (session-analyzer) -- thin wrapper that finds the JSONL and invokes this skill
3. An agent (pattern-reviewer) -- scans accumulated analyses for recurring patterns

The skill needs to be self-contained -- an agent invoking it provides only a JSONL path. The skill must handle all analysis logic, report writing, and pattern file management.
